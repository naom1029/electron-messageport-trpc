## `createElectronTRPCMain(opts)`

Connects a main-process router, or one or more typed registry channels, to
renderer windows. Each window's renderer is connected to its router, and the
connection is re-established on reload; windows that have already loaded when
registered are connected too. Returns an [`ElectronTRPCMainHandler`](#electrontrpcmainhandler).

For a single renderer-to-main router, use the default channel:

```typescript
import { createElectronTRPCMain } from 'electron-messageport-trpc/main';
import { appRouter } from './router';

createElectronTRPCMain({
  windows: [win],
  router: appRouter,
});
```

For multiple typed channels, pass a registry created by `defineElectronTRPC()`:

```typescript
import { createElectronTRPCMain } from 'electron-messageport-trpc/main';
import { electronTRPC } from './trpc';
import { appRouter } from './router';

createElectronTRPCMain({
  channels: electronTRPC,
  windows: [win],
  routers: {
    main: appRouter,
  },
});
```

`routers` is **partial**: this main process supplies routers only for the
channels it hosts. Other declared channels may be served by a utility process
(`createElectronTRPCUtility`) or brokered
(`createElectronTRPCRendererUtilityBridge`). Each supplied router must match the
router type registered for that channel, and every key must be a declared
channel.

### Parameters (single-router form)

```typescript
interface CreateElectronTRPCMainSingleOptions<TRouter, TWindow> {
  windows: readonly TWindow[];
  router: TRouter;
  createContext?: (opts: { window: TWindow }) => MaybePromise<unknown>;
  transformer?: DataTransformerOptions;
}
```

| Parameter | Type | Required | Description |
|---|---|---|---|
| `windows` | `readonly BrowserWindow[]` | Yes | Windows to wire to the default channel |
| `router` | `AnyRouter` | Yes | The main-process tRPC router |
| `createContext` | `(opts: { window }) => MaybePromise<unknown>` | No | Per-request context factory; receives the matching window |
| `transformer` | `DataTransformerOptions` | No | Overrides the router's configured data transformer |

### Parameters (registry form)

```typescript
interface CreateElectronTRPCMainOptions<TRegistry, TWindow> {
  channels: ElectronTRPCChannels<TRegistry>;
  windows: readonly TWindow[];
  // Partial: supply routers only for the channels this main process hosts.
  routers: Partial<{ [K in keyof TRegistry & string]: TRegistry[K] }>;
  createContext?: (opts: {
    window: TWindow;
    channel: keyof TRegistry & string;
  }) => MaybePromise<unknown>;
  transformer?: DataTransformerOptions;
  channelOptions?: Partial<
    Record<keyof TRegistry & string, { transformer?: DataTransformerOptions }>
  >;
}
```

| Parameter | Type | Required | Description |
|---|---|---|---|
| `channels` | `ElectronTRPCChannels<TRegistry>` | Yes | The `defineElectronTRPC()` registry |
| `windows` | `readonly BrowserWindow[]` | Yes | Windows to wire to the channels this process serves |
| `routers` | `Partial<{ [K in keyof TRegistry]: ... }>` | Yes | Routers for the channels **this main process serves**. A subset is allowed — other declared channels can be served by a utility process (`createElectronTRPCUtility`) or brokered (`createElectronTRPCRendererUtilityBridge`). Keys must be declared channels. |
| `createContext` | `(opts: { window, channel }) => MaybePromise<unknown>` | No | Per-request context factory; receives the window **and** the channel name |
| `transformer` | `DataTransformerOptions` | No | Default transformer for all channels |
| `channelOptions` | `Partial<Record<channel, { transformer? }>>` | No | Per-channel overrides; a channel's `transformer` takes precedence over the top-level `transformer` |

### `ElectronTRPCMainHandler`

The return value of `createElectronTRPCMain()`.

| Method | Description |
|---|---|
| `addWindow(window)` | Wires an additional window after creation (no-op if already wired or destroyed) |
| `removeWindow(window)` | Tears down one window: removes its `did-finish-load`/`closed` listeners and destroys its handlers |
| `destroy()` | Tears down every wired window |

```typescript
const handler = createElectronTRPCMain({ windows: [win], router: appRouter });

const second = new BrowserWindow({ /* ... */ });
handler.addWindow(second);

// later
handler.removeWindow(second);
handler.destroy();
```

## `createElectronTRPCUtilityClient(opts)`

Creates a typed tRPC client from the main process to one utility process.
Returns `{ client, destroy }` (not a bare client).

```typescript
const { client, destroy } = createElectronTRPCUtilityClient({
  channel: electronTRPC.worker,
  utility: child,
});

await client.render.mutate(input);

// when the utility is no longer needed
destroy();
```

It creates and wires the main-side port for you. (For manual port wiring, see
[`mainPortLink()`](#mainportlinkopts).)

### Ready handshake

The client waits for the utility process to be ready before it connects — the
library owns this handshake automatically (see
[`createElectronTRPCUtility()`](/reference/utility/)), so there is no manual
step. It also tears itself down when the utility process exits.

### Returns

| Field | Type | Description |
|---|---|---|
| `client` | `TRPCClient<RouterForChannel<TChannel>>` | The typed client for the channel's router |
| `destroy()` | `() => void` | Closes the kept main-process port and removes the `message`/`exit` listeners |

## `createElectronTRPCUtilityPool(opts)`

Creates typed clients for multiple utility process instances that expose the
same router type. Each instance uses the same ready-wait handshake and
`{ client, destroy }` machinery as `createElectronTRPCUtilityClient()`.

```typescript
const pool = createElectronTRPCUtilityPool({
  channel: electronTRPC.worker,
  utilities: {
    a: workerA,
    b: workerB,
  },
});

await pool.get('a').render.mutate(input);

// destroy every instance
pool.destroy();
```

The registry channel carries the router type. The pool key chooses the runtime
utility process instance.

### Methods

| Method | Description |
|---|---|
| `get(instance)` | Returns the typed `client` for that instance (throws on an unknown instance key) |
| `destroy()` | Destroys every instance's client (closes ports, removes listeners) |

## `createElectronTRPCRendererUtilityBridge(opts)`

Brokers a renderer-to-utility MessagePort. The main process creates the port
pair and transfers one side to the renderer and the other side to the utility
process.

```typescript
createElectronTRPCRendererUtilityBridge({
  window: win,
  channel: electronTRPC.worker,
  utility: child,
});
```

Like `createElectronTRPCMain()`, it connects each window (including already-loaded
ones) and re-connects on reload. The main process stays out of the tRPC request
path after the port is handed off.

## `mainPortLink(opts)`

Creates a tRPC client link for `MainPortLike`-compatible ports in the main process. Use this when the main process is the tRPC client, such as calling a utility process over `MessageChannelMain`.

### Parameters

```typescript
interface MainPortLinkOptions {
  port: MainPortLike | Promise<MainPortLike>;
  transformer?: DataTransformerOptions;
}
```

| Parameter | Type | Required | Description |
|---|---|---|---|
| `port` | `MainPortLike \| Promise<MainPortLike>` | Yes | The client-side main-process port. Electron's `MessagePortMain` satisfies `MainPortLike` |
| `transformer` | `DataTransformerOptions` | No | Data transformer matching the remote router |

### Example

```typescript
import { createTRPCClient } from '@trpc/client';
import { MessageChannelMain, utilityProcess } from 'electron';
import { mainPortLink } from 'electron-messageport-trpc/main';

const child = utilityProcess.fork('utility-worker.js');
const { port1, port2 } = new MessageChannelMain();

child.postMessage({ type: 'connect' }, [port1]);

const client = createTRPCClient({
  links: [mainPortLink({ port: port2 })],
});
```

<Aside type="tip">
For typed utility wiring, prefer `createElectronTRPCUtilityClient()`, which owns
the ready handshake and lifecycle for you. `mainPortLink()` remains available for
custom port wiring.
</Aside>

## `createPortBroker()`

Creates a port broker that generates MessagePort pairs for connecting renderers to the main process.

### Returns

A `PortBroker` object.

### `PortBroker`

#### `broker.createRendererPort(webContents, opts?)`

Creates a `MessageChannelMain` pair, sends one port to the specified renderer, and returns the other.

**Parameters:**

| Parameter | Type | Required | Description |
|---|---|---|---|
| `webContents` | `WebContents` | Yes | The target renderer's `webContents` object (e.g., `win.webContents`) |
| `opts` | `CreateRendererPortOptions` | No | Port options. `{ channel?: string }` — names the channel sent to the renderer, defaulting to `'default'`. Set this to target a specific typed channel when servicing multiple channels |

**Returns:**

```typescript
{
  serverPort: MessagePortLike;
}
```

The `serverPort` is the main-process side of the channel (a `MessagePortMain`, which satisfies `MessagePortLike`). Pass it to `createPortHandler()` to start handling tRPC requests.

### Example

```typescript
import { createPortBroker } from 'electron-messageport-trpc/main';

const broker = createPortBroker();

// After the window loads:
const { serverPort } = broker.createRendererPort(win.webContents);
```

<Aside type="note">
`createRendererPort` sends the port to the renderer; the preload script must call `exposePortReceiver()` to receive it.
</Aside>

## `createPortHandler(opts)`

Attaches a tRPC request handler to a MessagePort. It listens for incoming tRPC requests (queries, mutations, subscriptions) and routes them to the given router.

<Aside type="caution">
This is a low-level API. The supplied port should be dedicated to the electron-messageport-trpc protocol; do not multiplex app-defined `postMessage()` traffic on the same port. Messages that do not match the protocol are discarded.
</Aside>

### Parameters

```typescript
interface CreatePortHandlerOptions<TRouter extends AnyRouter> {
  port: MessagePortLike;
  router: TRouter;
  createContext?: () => MaybePromise<unknown>;
  transformer?: DataTransformerOptions;
}
```

| Parameter | Type | Required | Description |
|---|---|---|---|
| `port` | `MessagePortLike` | Yes | The server-side MessagePort (from `createRendererPort`) |
| `router` | `AnyRouter` | Yes | Your tRPC router instance |
| `createContext` | `() => MaybePromise<unknown>` | No | Factory function called for every request to create the context object |
| `transformer` | `DataTransformerOptions` | No | Overrides the router's configured data transformer |

### Returns

A `PortHandler` object.

### `PortHandler`

#### `handler.destroy()`

Stops all active subscriptions, aborts their `AbortController`s, and closes the underlying MessagePort.

Call this when the associated `BrowserWindow` is closed or the connection is no longer needed.

### Example

```typescript
import { createPortBroker, createPortHandler } from 'electron-messageport-trpc/main';
import { appRouter } from './router';

const broker = createPortBroker();

app.whenReady().then(() => {
  const win = new BrowserWindow({ /* ... */ });

  win.webContents.on('did-finish-load', () => {
    const { serverPort } = broker.createRendererPort(win.webContents);

    const handler = createPortHandler({
      port: serverPort,
      router: appRouter,
      createContext: async () => ({
        // per-request context
        windowId: win.id,
      }),
    });

    // Clean up when the window closes
    win.on('closed', () => {
      handler.destroy();
    });
  });
});
```

### Subscription Handling

When a `subscription` request arrives:

1. The procedure is called and must return an async iterable.
2. Each value it yields is delivered to the client.
3. If the procedure yields `tracked(id, data)`, the event ID is carried through to enable tRPC v11 subscription resumption.
4. When the client unsubscribes — or the port closes — the subscription is aborted automatically.
