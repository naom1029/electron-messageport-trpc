One of the key advantages of MessagePort over `ipcMain`/`ipcRenderer` is support for **flexible communication topologies**. The main process does not need to be a bottleneck.

## Renderer-to-Renderer Communication

Two renderer processes can communicate directly by having the main process broker a MessagePort pair between them.

### Main Process (Broker)

```typescript
import { BrowserWindow } from 'electron';
import { createPortBroker, createPortHandler } from 'electron-messageport-trpc/main';
import { appRouter } from './router';

const broker = createPortBroker();

// Create two windows
const winA = new BrowserWindow({ /* ... */ });
const winB = new BrowserWindow({ /* ... */ });

// When both windows are ready, create a port pair
// Window A acts as the "server", Window B as the "client"
winA.webContents.on('did-finish-load', () => {
  winB.webContents.on('did-finish-load', () => {
    // Create a port for window B (client side)
    const { serverPort } = broker.createRendererPort(winB.webContents);

    // Attach the tRPC handler to the server port
    // This handler runs in the main process but could also be
    // attached in window A's process
    createPortHandler({
      port: serverPort,
      router: appRouter,
    });
  });
});
```

<Aside type="note">
In this pattern, the main process still hosts the tRPC handler. For true renderer-to-renderer communication without the main process handling requests, you would need to pass both ports to the respective renderers -- this requires custom port forwarding logic.
</Aside>

## Utility Process Communication

Electron's [utility process](https://www.electronjs.org/docs/latest/api/utility-process) runs in a separate Node.js thread. You can connect it via MessagePort for offloading heavy computation.

### Utility Worker

```typescript
// utility-process.ts (runs in a utility process)
import { createParentPortHandler } from 'electron-messageport-trpc/utility';
import { utilityRouter } from './router';

createParentPortHandler({
  router: utilityRouter,
  parentPort: process.parentPort,
});
```

### Main to Utility

```typescript
// main process
import { createTRPCClient } from '@trpc/client';
import { MessageChannelMain, utilityProcess } from 'electron';
import { mainPortLink } from 'electron-messageport-trpc/main';

const child = utilityProcess.fork('utility-process.js');
const { port1, port2 } = new MessageChannelMain();

child.postMessage({ type: 'connect' }, [port1]);

const client = createTRPCClient({
  links: [mainPortLink({ port: port2 })],
});
```

### Renderer to Utility

```typescript
// main process broker
import { createPortBroker } from 'electron-messageport-trpc/main';
import { utilityProcess } from 'electron';

const child = utilityProcess.fork('utility-process.js');
const broker = createPortBroker();

win.webContents.on('did-finish-load', () => {
  const { serverPort } = broker.createRendererPort(win.webContents);
  child.postMessage({ type: 'connect' }, [serverPort]);
});
```

This keeps the main process out of the request path. It only creates and transfers the port pair.

### Architecture

```
┌─────────────┐   MessagePort   ┌──────────────┐   parentPort   ┌─────────────────┐
│   Renderer   │<===============>│  Main Process │<=============>│ Utility Process  │
└─────────────┘                 └──────────────┘                └─────────────────┘
```

## Choosing a Topology

| Topology | Use Case | Pros | Cons |
|---|---|---|---|
| Renderer to Main | Standard RPC, DB access, native APIs | Simple setup | Main process is the bottleneck |
| Renderer to Renderer | Shared state, collaboration features | Direct communication | Requires brokering |
| Main to Utility | Heavy computation, background tasks | Offloads main thread | Extra process overhead |
| Renderer to Utility | Direct offloading (via broker) | Best performance | Most complex setup |
